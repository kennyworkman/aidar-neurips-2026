Yes — the two categories are strong, but I would make the general theme slightly broader:

AI Data Readiness is about turning scientific data ecosystems into substrates that models can train on, agents can act on, and communities can evaluate.

The current proposal already supports this framing: it defines AIDaR around foundation models, lab-in-the-loop agents, digital twins, RAG, provenance, uncertainty, benchmark design, and operational readiness, rather than just “better datasets.”

I’d organize the theme into four categories:

1. AI-facing data infrastructure

Your first category is essential. I would phrase it less as storage and more as AI-operable scientific data systems.

This includes storing, tagging, linking, versioning, permissioning, and exposing experimental data so agents and foundation models can use it with context. For biology, this means assay metadata, protocols, batch structure, sample/donor relationships, instruments, controls, perturbations, negative results, units, data lineage, and access policies.

Better wording:

Data infrastructure for AI systems: How scientific data should be stored, annotated, versioned, governed, and exposed so agents and foundation models can retrieve, reason over, and act on experimental evidence with provenance.

2. Semantic and experimental context

I would add this as its own category because it is the bridge between raw infrastructure and useful intelligence. Agents do not just need files; they need to know what the data means.

For molecular foundation models, this is especially important: sequence, structure, perturbation, assay, phenotype, dose, time, cell type, tissue context, and experimental design all have to be represented in ways the model can learn from or reason over. For agents, this includes knowing what comparison is valid, what unit is independent, which metadata field is biological versus technical, and what provenance chain supports a conclusion.

Possible wording:

Semantic readiness and experimental context: Representing the biological, physical, and experimental meaning of data so AI systems can distinguish valid evidence from artifacts, metadata leakage, batch effects, and unsupported inference.

3. Benchmarks and evals for AI behavior

Your second category is also essential. I would make the key point that evals are not just leaderboards; they are measurement instruments for whether data is usable by moving frontier systems.

This category can include benchmark construction, task design, ground-truth curation, agent trajectories, data ablations, provenance stress tests, agent failure analysis, and evals that connect data interventions to downstream scientific behavior.

Better wording:

Benchmarks and evaluations for AI data readiness: Constructing evaluation sets that measure whether agents and foundation models can use scientific data to perform practical work, and whether improvements in data structure, metadata, provenance, or context actually improve downstream behavior.

4. Governance, provenance, and deployment feedback

Some of this fits under infrastructure, but I think it deserves explicit visibility because it makes the theme feel consequential beyond model performance. This covers auditability, compliance, reproducibility, permissions, licensing, human review, observability, and feedback from deployed systems.

For agents, the important thing is not just “can the model answer?” but “can we reconstruct what data it used, what transformations it performed, why it reached a conclusion, and whether the result is reproducible?”

Possible wording:

Provenance, governance, and feedback loops: Capturing how data, tools, models, and human decisions interact in deployed scientific AI systems, with enough lineage and observability to support reproducibility, compliance, and continuous improvement.

A compact final taxonomy could be:

AI-operable data infrastructure — storage, metadata, APIs, access, versioning, workflow exposure.
Semantic and experimental context — assay meaning, biological units, protocols, controls, uncertainty, negative results.
Benchmarks and evals for AI behavior — measuring whether agents and molecular foundation models can use data to do real scientific work.
Provenance, governance, and deployment feedback — auditability, compliance, reproducibility, observability, and learning from real use.

I would not make the theme only “data infrastructure + evals.” That is close, but it skips the crucial middle layer: scientific meaning. The strongest version is: infrastructure makes data available, semantics makes it meaningful, evals measure whether AI can use it, and provenance/governance make the system trustworthy.

The Nature paper explicitly describes heterogeneous inputs being encoded into a universal 3D latent representation, then decoded back into physical predictions.
